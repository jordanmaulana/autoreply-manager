# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

Backend uses `uv`, frontend uses `pnpm`. Most tasks are in the Makefile:

```bash
make dev        # Django dev server on :8000
make lint       # ruff format . && ruff check . --fix
make db         # start pgvector container (host port 15432, app/app/app)
make mmg        # makemigrations
make migrate    # migrate
make poller     # manage.py poll_threads (Threads reply polling loop)
make web        # React dev server (frontend/, port 5173)
make tw-run     # Tailwind watcher for Django templates (static/input.css → static/output.css)
make tw-build   # one-shot version of tw-run
make dock       # docker compose down/build/up using .env.docker
make upgrade    # uv sync + lock upgrade
make audit      # pnpm audit fix in frontend/
```

Frontend (inside `frontend/`): `pnpm dev`, `pnpm build` (tsc -b + vite build), `pnpm lint` (eslint).

There is no test suite configured. Ruff config lives in `pyproject.toml` (line length 100, py310, rules E/F/I/B/UP).

API smoke-test `.http` files live in `_rest/` (committed) and `.rest.local/` (local only). `_docs/idea.md` is the product brief.

## Environment

Copy `.env.example` to `.env` (loaded by `core/settings.py` via `load_dotenv(override=True)` — **the `.env` file beats real environment variables**, which surprises in Docker/CI).

The database is **always Postgres with pgvector** (`pgvector/pgvector:pg16`; pgvector is required for RAG). `make db` maps the container to host port **15432** (5432/5433 usually taken by native Postgres) and `.env.example` matches — but `settings.py` falls back to `5432` when `POSTGRES_PORT` is unset, so exporting env vars by hand instead of copying `.env.example` silently connects to the wrong Postgres. Docker compose uses `.env.docker` with `POSTGRES_HOST=postgres`.

Feature env: `OPENAI_API_KEY` (embeddings + replies), `INSTAGRAM_APP_ID/SECRET`, `THREADS_APP_ID/SECRET`, `WHATSAPP_APP_SECRET`, `META_WEBHOOK_VERIFY_TOKEN`. `SITE_URL` must be a public HTTPS URL for Meta webhooks/OAuth callbacks (ngrok in dev).

Frontend vars (`VITE_API_URL`, `VITE_GOOGLE_CLIENT_ID`) live in the **root** `.env.example`, not in `frontend/` — Vite only auto-loads `frontend/.env`, so locally they reach the SPA via the Vite dev proxy (`/api` → `:8000`, so `VITE_API_URL` can stay empty) and in production via Docker build args.

### Fresh-clone traps

- **`.env.docker` is gitignored and has no example file.** Every docker path (`make dock`, `update.sh`) fails until you hand-write it from `.env.example`.
- `static/output.css` is gitignored — the Django dashboard renders unstyled until `make tw-build`.
- `frontend/src/routeTree.gen.ts` is gitignored — generated by the Vite plugin, so `tsc -b` fails until `pnpm dev`/`pnpm build` has run once.

## Architecture

Two separate frontends share one Django backend:

1. **React SPA** (`frontend/`) — the user-facing app. React 19, Vite, TanStack Router (file-based routes in `src/routes/`, route tree auto-generated), TanStack Query, jotai, Tailwind v4. Talks to the backend over `/api/v1/` with DRF token auth.
2. **Django templates** (`templates/` + `static/`) — superuser-only shell at `/dashboard/` (session auth, `SuperuserRequiredMixin` in `core/views.py`), with its own Tailwind build (`make tw-run`) independent of the SPA's. **`/dashboard/` is an empty placeholder** (renders a welcome line, no data). The real operator UI is **Django admin at `/admin/`**, where all five models are registered (`core/admin.py`).

### Backend layout

- `core/` is both the Django project (settings, urls, wsgi) and the single app (models, views, admin, payments). `api/` holds only DRF endpoint modules and is deliberately **not** an installed app (no models).
- Endpoints are function-based DRF views in `api/v1/<domain>_api.py`, wired in `api/v1/urls.py`. Default policy is TokenAuthentication + IsAuthenticated; open endpoints opt out with `@permission_classes([AllowAny])`. Ownership is enforced ad-hoc per view (`filter(actor=request.user)`, or `account__actor=request.user` for `ReplyLog`) — there is no object-permission class. **No throttling anywhere**, including on the AllowAny auth endpoints.
- **Auth**: two ways in. Google — SPA sends an ID token to `POST /api/v1/auth/google/`, `GoogleAuthSerializer` verifies it (google-auth, requires `email_verified`), user is get-or-created by email, DRF token returned. Email/password — `auth/register/` + `auth/login/` (both AllowAny). `register` refuses existing emails rather than setting a password (there is no email verification, so this would hijack Google-created accounts); `login` returns one generic error for every failure mode to avoid enumeration. The frontend stores the token in localStorage and sends `Authorization: Token <key>` (`frontend/src/lib/api.ts`); auth code lives in `frontend/src/features/auth/`.
- **Models** (`core/models.py`): subclass `BaseModel` — string PK from `bson.ObjectId`, plus `created_on`/`updated_on`/`actor`. Subclasses must declare `class Meta(BaseModel.Meta): abstract = False` and re-add their own constraints. `AppSetting` is the exception: a plain `models.Model` typed key-value store with a normal auto PK; it, and `AppSetting.get()`, are currently unused.
- **Payments**: Mayar client in `core/payments/mayar.py`. Only `PAID_STATUSES` and `verify_webhook` have callers; the webhook at `/api/v1/payments/mayar/webhook/` is a stub. **There is no subscription/plan model in the schema** — nothing to activate yet, and nothing in the app gates on payment.

### Auto-reply / RAG

Pipeline: chunking (tiktoken, `core/rag/chunking.py`, 500 target / 800 max / 60 overlap tokens) → OpenAI embeddings (`embeddings.py`, 1536 dims) → pgvector cosine retrieval → gpt-4o-mini reply (`answer.py`: `TOP_K=5`, `MAX_DISTANCE=0.6`, per-platform `REPLY_CHAR_LIMITS` of IG 800 / WA 3000 / Threads 480). `inbound.py` is the end-to-end handler.

**Nothing starts at boot** — there is no `apps.py` and no `AppConfig.ready()`. Both background paths are per-request fire-and-forget `daemon=True` threads:

- Embedding — `start_embedding_job` (`core/rag/runner.py`), spawned from three call sites in `knowledge_api.py` (create, PUT when the description changed, rebuild). Drives a pending/processing/ready/failed status machine on `Knowledge` via an atomic conditional UPDATE (0 rows claimed → 409).
- Inbound replies — `api/v1/webhooks_api.py` spawns processing after the signature check, so the webhook 200s before the reply is generated.

Consequences of having no celery, both worth knowing before debugging a "stuck" record: the 15-minute stale-claim window is **passive** — it only lets a *new* `start_embedding_job` re-claim, and no sweeper re-drives stuck PROCESSING/PENDING rows, so a crash mid-embed needs a manual `/rebuild/` after 15 minutes. Likewise a gunicorn restart loses an in-flight reply thread, and Meta's retry is a silent no-op because the `ReplyLog` row already exists.

Vector search depends on migration `0004_kchunk_hnsw_index` (raw SQL, `vector_cosine_ops`, `m=16, ef_construction=64`) — changing `EMBEDDING_DIMENSIONS` or the distance metric means editing that migration. `0002_vector_extension` runs `CREATE EXTENSION vector`, which needs superuser-ish DB privileges (fine on the pgvector image, the first thing to fail on managed Postgres).

### Social platforms

`core/social/` holds per-platform clients over Graph `v23.0` (`meta_common.py`), plus `get_sender(platform)` as the dispatch point.

- **Instagram** — DMs via signed Meta webhook at `/api/v1/webhooks/instagram/`. Uses Instagram Login (`graph.instagram.com`), not Facebook Login.
- **WhatsApp** — Cloud API, signed webhook at `/api/v1/webhooks/whatsapp/`. Connects via pasted WABA credentials; Embedded Signup deliberately skipped.
- **Threads** — no DM API and no reply webhooks at MVP, so auto-reply targets replies to the user's own posts, discovered by the `poll_threads` command (no cursor by design — the idempotency guard covers re-polling). Sending is two-step (create + publish). Quirk: Threads `sender_id` is the author's **username**, not a numeric id.

Cross-cutting rules:

- One `SocialAccount` links to one `Knowledge`; a `Knowledge` is shared by many accounts.
- **`ReplyLog`'s unique `(account, platform_message_id)` is the idempotency guard** — an `IntegrityError` catch in `core/rag/inbound.py`, covering webhook retries, multi-worker races, and Threads re-polls. Never dedupe in memory.
- `verify_signature` (X-Hub-Signature-256, HMAC over the raw body) **fails closed** when the app secret is unconfigured — that's why webhooks 401 locally before you set it.
- IG/Threads OAuth callbacks recover the user from a signed `state` (TimestampSigner, **10-minute** max age — the cause of "link expired" bounces), and are AllowAny.
- `manage.py refresh_tokens` must run daily in production (IG/Threads tokens last ~60 days; it refreshes inside a 10-day window).
- Access tokens are stored **plaintext** — never expose them in serializers or admin `list_display`.

### Frontend conventions

- `features/<domain>/` slices are uniform: `api.ts` (thin fns over the `api<T>()` wrapper), `types.ts`, `hooks.ts` (TanStack Query), `components/`. Query keys are plain inline arrays, no factory. Paths end in a **trailing slash** (DRF).
- **Route protection is not `beforeLoad`** (used nowhere) — `__root.tsx` renders `<AuthGate>` (`features/auth/components/auth-gate.tsx`), which does component-level `useEffect` redirects. Every route file is unprotected on its own; add public paths to `SPLASH_PATHS`.
- jotai holds exactly two atoms (`tokenAtom`, `userAtom` in `features/auth/state.ts`). Everything else is TanStack Query server state or local `useState`. `lib/api.ts` reads the token from localStorage per call, not from the atom.
- **No shadcn component is vendored and `radix-ui` is never imported**, despite both being dependencies and `components.json` existing. All UI is hand-written Tailwind utility classes + lucide icons + the `cn()` helper. `src/components/ui/` does not exist yet; `npx shadcn add` would create it.
- react-toastify is the app-wide error/success surface (wired in `main.tsx`).
- `@/` → `./src` is declared in three places that must stay in sync: `vite.config.ts`, `tsconfig.json`, `tsconfig.app.json`.
- `frontend/README.md` is the **stock Vite template README** — not documentation for this project.

### Deployment

Single backend image (`Dockerfile.backend`): a node stage builds the template Tailwind CSS, then a python/uv stage runs gunicorn with whitenoise serving static files. `docker/backend-entrypoint.sh` waits for Postgres, then runs migrate + collectstatic **only when `ROLE=web`**. Compose has three services: `postgres`, `backend` (`ROLE=web`), and `threads-poller` (`ROLE=poller`, same image, runs `poll_threads`, skips migrate so the two containers don't race). `update.sh` is the pull-build-migrate deploy script. The React SPA is not part of this image and deploys separately.
